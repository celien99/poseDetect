# 当前项目落地实施建议

本文档面向当前项目的实际需求，目标不是“先做模型训练”，而是先把固定机位下的动作检测闭环跑通。

## 1. 总体原则

当前项目更合理的推进顺序是：

1. 先跑通推理
2. 再校准规则
3. 再做现场验证
4. 最后再决定是否需要训练

不建议一开始就把主要精力投入到训练，因为当前主路线本身是：

- 通用姿态模型
- 固定区域标定
- 规则动作识别
- 状态机流程判断

## 2. 第一阶段：先跑通单路视频方案

这一阶段的目标是：

- 能稳定读取视频或相机流
- 能输出动作 JSON
- 能输出标注视频
- 能看到流程 `OK / NG`

建议步骤：

1. 准备一个可用的姿态模型权重，例如 `yolo26n-pose.pt`
2. 准备一段真实或接近真实场景的视频
3. 复制示例配置：
   - [runtime.video.example.json](/Users/yyh/code/poseDetect/docs/runtime.video.example.json)
4. 修改以下配置：
   - `inference.source`
   - `inference.pose_model_path`
   - `inference.seat_regions`
   - `rules.actions`
   - `inference.state_machine.steps`
5. 执行推理：

```bash
python -m seat_inspection infer --config docs/runtime.video.example.json
```

这一阶段不追求高精度，先确认链路完整。

## 3. 第二阶段：完成区域标定

当前项目里，`seat_regions` 是固定机位方案的核心输入。

建议做法：

1. 使用代表性视频帧或相机画面做标定
2. 标定三个区域：
   - `overall`
   - `side_surface`
   - `bottom_surface`
3. 将标定结果保存为配置文件的一部分
4. 同一工位、同一机位尽量固定使用同一套区域参数

如果需要交互式标定，可使用：

```bash
python -m seat_inspection calibrate-regions --source "your_video_or_camera_source"
```

## 4. 第三阶段：调动作规则

当前项目动作识别的关键不是训练，而是调 `rules.actions`。

优先调这些参数：

- `hold_frames`
- `wrist_margin`
- `min_wrist_count`
- `lift_ratio_threshold`
- `reach_ratio_threshold`

建议方法：

1. 先只保留两个基础动作：
   - `touch_side_surface`
   - `lift_seat_bottom`
2. 用 5 到 10 段真实视频反复看 JSON 和标注视频
3. 先压低漏报，再逐步减少误报
4. 每次只调整一个参数，不要同时大改

## 5. 第四阶段：调流程状态机

当单动作识别基本稳定后，再调 `state_machine.steps`。

重点配置：

- `action`
- `min_frames`
- `require_all_steps`

建议做法：

1. 先只定义最小流程
2. 确保步骤顺序和现场 SOP 完全一致
3. 每一步的 `min_frames` 先保守设小
4. 确认 `OK / NG` 判断逻辑符合业务预期

## 6. 第五阶段：现场验证

现场验证阶段的目标是确认：

- 通用 pose 模型是否够用
- 固定 ROI 是否稳定
- 规则是否足够表达动作
- 状态机是否符合 SOP

建议至少覆盖：

- 正常样本
- 漏做步骤样本
- 错误顺序样本
- 动作不标准样本
- 遮挡样本
- 光照变化样本

建议记录：

- 误报场景
- 漏报场景
- 原因分类
- 对应视频片段

## 7. 什么时候再启动训练

只有在下面几类问题明显存在时，再建议投入训练：

### 7.1 姿态关键点不稳定

例如：

- 手腕点经常漂移
- 关键点频繁缺失
- 被设备遮挡后恢复很差

### 7.2 规则表达不了业务动作

例如：

- 两个动作的差异无法靠区域触达区分
- 需要识别复杂时序行为
- 动作标准和非标准之间差异过细

### 7.3 多工位、多机位差异过大

例如：

- 不同工位座椅位置变化大
- 不同班组动作姿态差异明显
- 统一阈值很难兼容

## 8. 训练阶段建议

如果后续确认必须训练，建议按下面顺序做：

1. 先用 `collect` 采集真实场景数据
2. 人工复核或修正自动伪标签
3. 形成稳定的 `train/val` 数据集
4. 再启动 `train`
5. 用同一批验证视频回归推理效果

不要在没有真实现场样本和验证流程时直接训练。

## 9. 当前推荐的最小落地路径

如果目标是尽快出一个可用版本，建议最小路径如下：

1. 使用现成 pose 权重
2. 只做单路视频或单路相机
3. 只保留两个基础动作
4. 只做最小 SOP 状态机
5. 用 10 段左右真实视频调规则
6. 通过后再考虑多相机
7. 最后再决定是否训练

## 10. 一句话结论

当前项目最合理的实施顺序不是“先训练模型”，而是：

**先把固定机位规则方案跑通，再根据现场误报、漏报和关键点稳定性决定是否训练。**
